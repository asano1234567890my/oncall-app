# 最適化ロジック（OnCallOptimizer）

ファイル: `backend/services/optimizer.py`

## 概要
Google OR-Tools の **CP-SAT ソルバー**を使って当直スケジュールを自動生成する。
ハード制約（絶対条件）とソフト制約（重み付きペナルティ）の2層構造で最適解を探索する。

---

## スコアリング（シフト種別ごとの負荷スコア）

| シフト種別 | スコア |
|---------|------|
| 日直（日曜・祝日の午前） | 0.5 |
| 当直・夜間（平日） | 1.0 |
| 当直・夜間（土曜） | 1.5 |
| 当直・夜間（日曜・祝日） | 1.0 |

**スコア範囲目安:** 0.5 〜 4.5（月間）

---

## ハード制約（HardConstraints）

違反した場合は解なし（infeasible）になる絶対条件。

### UIに公開中の設定

| 制約名 | キー | デフォルト | UI上の場所 |
|---------|------|-----------|-----------|
| 勤務間隔 | `interval_days` | 4日 | 基本ルール |
| 土日祝の合算勤務上限 | `max_weekend_holiday_works` | 3回 | 基本ルール |
| 土曜当直上限 | `max_saturday_nights` | 1回 | 基本ルール |
| 日祝シフトモード | `holiday_shift_mode` | `"split"` | 基本ルール |

### UIから非公開（バックエンドデフォルトで動作）

| 制約名 | キー | デフォルト | 非公開の理由 |
|---------|------|-----------|------------|
| 日祝・日直上限 | `max_sunhol_days` | 2回 | `max_weekend_holiday_works` に包含される |
| 日祝勤務上限 | `max_sunhol_works` | 3回 | 同上 |
| 日祝連続勤務回避 | `prevent_sunhol_consecutive` | true | `holiday_shift_mode` に連動（split→自動ON、combined→自動OFF） |
| 不可日をハード制約にする | `respect_unavailable_days` | true | 常にハード制約として運用する方針に決定 |
| ロック済みシフト | — | — | フロントエンドで固定したシフトは変更不可（常に有効） |

> **回数上限のUIは最小値1**（0にすると「配置不可」になり解なしリスク）。`interval_days` のみ0（制限なし）を許可。

## 日当直モード（`holiday_shift_mode: "combined"`）

日曜・祝日に「日直」と「当直」を同一人物が担当するモード。

| モード | 日祝スロット | スコア |
|-------|-----------|------|
| `split`（デフォルト） | 日直（0.5pt）+ 当直（1.0pt）を別人 | 合計1.5pt（2名） |
| `combined` | 日当直（1.5pt）を1名が担当 | 1名に1.5pt |

- combined モードでは `day_shift == night_shift`（同一医師ID）でスケジュールを返す
- フロントエンドの「日当直」バッジで視覚的に識別
- DB保存時は `day` と `night` の2レコードとして同じ医師IDで保存（スキーマ変更不要）

## ソフト制約と重み（ObjectiveWeights）

ペナルティとして目的関数に加算される条件。値が大きいほど優先度が高い。

### アクティブなウェイト

/dashboard のUIでは2軸のグループスライダーで操作する。

**軸1: 目標スコアへの近さ**（primaryKey: `target`）

| パラメータ名 | デフォルト | グループ比率 | 意味 |
|---------|------|------|------|
| `target` | 10 | 1.0 | 各医師の目標スコアとの差（絶対値）をペナルティ |
| `score_balance` | 30 | 0.3 | 全医師の（当月+過去累積）スコアの最大-最小差をペナルティ |

**軸2: 土日祝の均等化**（primaryKey: `sunhol_fairness`）

| パラメータ名 | デフォルト | グループ比率 | 意味 |
|---------|------|------|------|
| `sunhol_fairness` | 200 | 1.0 | 当月の日祝シフト回数の最大-最小差をペナルティ |
| `sat_month_fairness` | 100 | 1.0 | 当月の土曜当直回数の最大-最小差をペナルティ |
| `past_sunhol_gap` | 5 | 0.5 | 過去数か月の累積日祝回数差をペナルティ |
| `past_sat_gap` | 10 | 0.5 | 過去数か月の累積土曜回数差をペナルティ |

### 不活化中のウェイト（デフォルト0）

将来のバックエンド再設計で廃止または置き換え予定。

| パラメータ名 | 旧デフォルト | 不活化理由 | 将来の方針 |
|---------|------|------|------|
| `month_fairness` | 100 | 目標スコア方式に統合（`target` で代替） | 廃止 |
| `gap5` | 100 | 理想間隔方式に置き換え予定 | 再設計 |
| `gap6` | 50 | 同上 | 再設計 |
| `sat_consec` | 80 | 加重累積均等化に置き換え予定 | 再設計 |
| `sunhol_3rd` | 80 | ハード連動ペナルティに置き換え予定 | 再設計 |
| `weekend_hol_3rd` | 0 | 同上 | 再設計 |
| `soft_unavailable` | 1000（固定） | 設定UI非表示・optimizer内部固定 | — |

### 不可日の扱いについて

- 不可日は常にハード制約（`respect_unavailable_days` は常にtrue）
- 解なし時の最終手段として、管理者が個別不可日を「ソフト化」できる（`is_soft_penalty: true`）
- ソフト化された不可日のペナルティは内部固定値1000（他の重みの5〜10倍、ほぼハード制約に近い挙動）
- 生成成功時、ソフト不可日が違反された場合はフロントエンドでtoast通知（医師名・日付・シフト種別を表示）

## 将来の再設計計画（V3 Optimizer）

### 設計方針

**「ハード制約に追従するソフト制約」** — ハードが上限を決め、ソフトがその範囲内で最適配分する。ユーザーが個別ウェイトを調整する必要をなくす。

### 1. 加重累積均等化（土日祝の均等化を置き換え）

現行の `sunhol_fairness` + `sat_month_fairness` + `past_*_gap` + `sat_consec` + `*_3rd` 系を**1つの仕組み**に統合。

**計算方法:**
```
加重カウント[医師] = 当月の回数 × 1.0
                   + 先月の回数 × decay
                   + 2か月前の回数 × decay²
                   + ...
                   + Nか月前の回数 × decay^N
                   + (N+1)か月前以降 = 0

ペナルティ = max(加重カウント) - min(加重カウント)
```

- `decay` はユーザーに見せず内部固定（例: `1/N` の線形減衰で N+1 か月前に自然に0）
- ユーザー設定は「何か月前まで考慮するか」の1つだけ
- カテゴリ別（土曜当直・日祝・合算）に独立して計算
- **重要: 差分は符号付きで累積すること**。絶対値で毎月累積すると、1月に多く働いて2月に少なく調整した医師が「両月とも悪い」扱いになり、長期的な公平性が壊れる
- **重要: 新規赴任医師の過去スコア補正が必要**。現行の`apply_history_score_baseline()`（`optimizer_history.py`）は履歴がない医師に経験者の平均スコアを代入している。加重累積では月ごとの回数を扱うため、「いなかった月は平均値で埋める」か「計算対象期間を在籍期間に短縮する」かを再設計する必要がある

**UI設定:**
```
土曜当直の均等化  [スライダー 0-200]
土日祝の均等化    [スライダー 0-200]
均等化の考慮期間  [3か月前まで ▼]
```

**置き換え対象:**
- `sunhol_fairness` → 土日祝の均等化
- `sat_month_fairness` → 土曜当直の均等化
- `past_sunhol_gap` → 考慮期間に統合
- `past_sat_gap` → 考慮期間に統合
- `sat_consec` → 加重累積で自然に回避（先月多い人は今月減る）
- `sunhol_3rd` → 均等化で自然に分散
- `weekend_hol_3rd` → 同上

### 2. 理想間隔方式（勤務間隔のゆとりを置き換え）

現行の `gap5` + `gap6`（固定2段階）を**グラデーション方式**に置き換え。

**計算方法:**
```
ハード間隔: interval_days（例: 4日）
理想間隔: ideal_gap（例: 7日）

各医師の連続勤務ペアごと:
  actual_gap = 次の勤務日 - 前の勤務日
  if actual_gap < ideal_gap:
    penalty += (ideal_gap - actual_gap)  # 近いほど重い
```

- ユーザー設定は「理想間隔」の1つだけ（ハード+ゆとり日数）
- ハードとの間が自動的にグラデーション（段階数はハードと理想の差で決まる）

**UI設定:**
```
勤務間隔（ハード）  [4日]
理想間隔            [7日]  ← ソフト。できればこれ以上空ける
```

**置き換え対象:**
- `gap5` → 自動計算
- `gap6` → 自動計算

### 3. 累積目標乖離（目標スコア近似を強化）

現行の `target`（当月のみ）+ `score_balance`（累積公平性）を**累積目標乖離**に統合。

**計算方法:**
```
累積実績[医師] = 当月スコア + 過去累積スコア
累積目標[医師] = 月数 × 月ごとの目標値
累積差分[医師] = 累積実績 - 累積目標  ← 符号付きで保持
乖離 = |累積差分[医師]|              ← 最後に絶対値をとる

ペナルティ = Σ 乖離[医師]
```

- **重要: 各月の差分を絶対値で足し上げてはいけない**。符号付きで累積し、ペナルティ計算時に初めて絶対値をとる。これにより「先月多く働いた→今月少なめ」が正しく相殺される
- 当月だけの `target` は従来通り残す（短期バランス用）
- `score_balance`（全員の累積max-min差）→ 累積目標乖離に置き換え
- `month_fairness`（当月max-min差）→ 廃止（`target` で代替）

**置き換え対象:**
- `score_balance` → 累積目標乖離
- `month_fairness` → 廃止済み

### 4. 不可日のソフト化（実装済み）

**現在の実装:**
- 解なし時に管理者が個別不可日を「ソフト化」（DoctorManageDrawer の解なし対策セクション）
- ソフト化された不可日は内部固定ペナルティ1000（設定UI非表示）
- 生成後、違反があればtoast通知で「誰の何日が守れなかったか」表示
- 将来的に医師が自分で「できれば避けたい」を選択可能にする余地あり

### 5. 解なし診断の強化（実装済み）

**診断フロー:**
1. **Phase 1（MUS検出）**: CP-SAT AddAssumptionsで最小矛盾制約群を特定
2. **Phase 2a（管理者設定探索）**: 4設定（勤務間隔/土日祝合算上限/土曜当直上限/スコア上限）を単独→2設定の組み合わせで探索。実質無意味な値（間隔0日、上限≧月内日数等）は除外
3. **Phase 2b（不可日最小解除セット）**: 競合不可日に「残す/外す」変数を付けて外す数を最小化。「この組み合わせを全部解除すれば確実に解ける」を検証済みで返す
4. **人手不足日検出**: MUSで原因特定不能な場合のフォールバック
5. **案内モーダル**: 検証済み解決策がない場合、設定パネル・医師管理ドロワーへの誘導を表示

**不活化した制約:** max_sunhol_days / max_sunhol_works はデフォルトNull（UIから廃止済み、max_weekend_holiday_worksでカバー）

**Gemini連携:** 現在スキップ（ソルバー側の検証済み結果が十分なため）。コードは保持しており将来再有効化可能。

### 将来の基本ルールUI（全設計完了後）

```
勤務間隔           [4日]          ← ハード
理想間隔           [7日]          ← ソフト
土曜当直上限        [1回]          ← ハード
土日祝合算上限      [3回]          ← ハード
均等化の考慮期間    [3か月前まで]    ← 加重累積の範囲
日祝シフトモード    [別々/日当直]
シフトスコア + スコア範囲
```

上級設定の「優先度の調整」スライダーは原則不要になる（ハード設定に追従してソフトが自動生成されるため）。

## 主なメソッド

| メソッド | 役割 |
|---------|------|
| `build_model()` | CP-SATモデルを構築（制約・変数・目的関数を設定） |
| `solve()` | ソルバーを実行して最適解を返す |
| `is_holiday(date)` | 指定日が祝日かどうかを判定 |
| `is_sunday(date)` | 指定日が日曜日かどうかを判定 |

## 履歴スコア計算（`optimizer_history.py`）

| 関数名 | 役割 |
|---------|------|
| `build_past_total_scores()` | DBの過去シフト履歴から医師ごとの累積スコアを計算 |
| `score_historical_shift()` | 個別のシフトのスコアを計算 |

過去スコアは最適化時に参照され、長期的な公平性を保つために使用される。

## 最適化リクエスト（OptimizeRequest）

フロントエンドから送られる主なパラメータ：

```
- target_year_month: 対象年月
- doctors: 医師リスト（id, name, スコア設定, 不可日等）
- locked_shifts: 固定シフト一覧
- holidays: 祝日リスト
- hard_constraints: ハード制約設定
- objective_weights: ソフト制約の重み設定
```
