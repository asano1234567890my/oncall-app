## 全体設計思想 — 3つの独立した拡張軸

全要件を分析した結果、将来の拡張は **3つの独立した軸** で整理できる。
これらは互いに独立して開発でき、組み合わせると掛け算で機能が広がる。

```
軸①「枠」── 1日に何人配置するか、枠ごとにどんなルールか
  現在: 1日1人固定
  → 枠番号でメイン枠/サブ枠を分離（サブ枠は独自ルール）

軸②「属性」── 医師にタグをつけて、タグごとに制約を変える
  現在: 全医師に同一ルール
  → 診療科・職位などの属性で制約を個別化

軸③「シフト種別」── 当直以外の勤務種別を追加する
  現在: 日直/当直の2種類固定
  → ICU当直、オンコール、3交代など自由定義
```

**この3軸の関係:**

| 組み合わせ | 実現できること |
|-----------|--------------|
| 枠のみ | 2人体制の手動配置。専攻医を手動でサブ枠に入れる |
| 属性のみ | 科ごとに間隔や回数上限を変える。科バランスを保つ |
| シフト種別のみ | ICU当直・救急当直を別スコアで管理 |
| **枠 × 属性** | 「サブ枠には専攻医だけ」「メイン枠は上級医 or 病棟医」 |
| **属性 × シフト種別** | 「泌尿器科はオンコールのみ」「外科は当直+ICU当直」 |
| **枠 × シフト種別** | メイン枠は当直、サブ枠はオンコール（異なるルール） |
| **3軸全部** | フル機能。ナース版もこの延長 |

**重要な原則: 枠をまたいだ制約は交差させない。**
メイン枠の制約とサブ枠の制約は完全に独立。
「メインが上級医の日だけサブを有効化」のような枠間の依存は**UIの制限**（ドロップ可否）で表現し、自動生成エンジンの中には持ち込まない。
これにより各軸の実装を単純に保てる。

## 軸① 枠番号（メイン枠 / サブ枠）

### 枠番号方式とは

当直記録テーブルに「枠番号」（1=メイン、2=サブ）を1つ追加する方式。

**基本ルール:**
- メイン枠（枠1）: 自動生成の対象。既存の制約チェック・スコア計算が適用される
- サブ枠（枠2）: D&D手動配置 or 自動生成（2回目のソルバー実行）。枠2独自の制約を適用
- メイン枠とサブ枠の制約は交差させない（独立した2つのルール体系）
- サブ枠の自動生成は「既存のソルバーを2回目に走らせる」ことで実現（新しい仕組み不要）
- 「確定→仮保存にコピー」「確定→再編集」ではサブ枠は自動で外れる

**医師側の対応:**
- 医師登録時に「メイン枠対象」か「サブ枠専用」かを設定可能
- サブ枠専用の医師（専攻医など）は自動生成から除外されるが、D&D画面には表示される
- メイン枠対象の医師はサブ枠にも配置可能（兼務OK）

### [E] 枠番号基盤 + 2人体制（軸①の基盤）

- **ニーズ**: 同じ日に2人配置したい。特に上級医+専攻医のペア当直
- **現状の限界**: 全システムが「1日1人」前提。記録・画面・集計すべてが1人で設計されている

**段階的アプローチ:**

#### Step 1: 枠番号+手動配置 ★★☆☆☆〜★★★☆☆

既存の自動生成は一切変えず、D&D編集画面でサブ枠の操作を可能にする。

- **当直記録テーブル**: 「枠番号」列を1つ追加（初期値=1。既存データはすべてメイン扱い）
- **医師テーブル**: 「サブ枠専用」フラグを追加（外部医師フラグと同じ仕組み）
- **自動生成**: 「サブ枠専用」の医師を除外する処理だけ追加（既存の外部医師除外と同じパターン）
- **D&D編集画面**:
  - セルを上下2段に分割（上段=メイン枠、下段=サブ枠）
  - パレットの医師リストを「メイン医師」「サブ専用」の2グループで表示
  - サブ専用医師はサブ枠にだけドロップ可能
  - メイン医師はどちらにもドロップ可能
- **サブ枠の制約チェック**: 枠2専用の簡易ルール（連続勤務OK、間隔○日、月の上限N回）
  メイン枠の制約とは完全に独立した設定画面
- **確定表・医師向け当直表**: 両方の枠を表示
- **仮保存コピー・再編集時**: メイン枠だけコピー（サブ枠は自動で外れる）
- **レポート・PDF出力**: 初期はメイン枠のみ対象。サブ枠の集計は後から追加
- **工数**: 5〜10日

#### Step 2: サブ枠の自動生成（ソルバー2回実行方式） ★★★☆☆

メイン枠を先に自動生成 → メイン枠の結果から「サブ枠が必要な日」を自動判定 → 同じソルバーを制約と医師を差し替えて2回目実行。

**既存の仕組みだけで実現できる理由:**

| 既存機能 | サブ枠生成での使い方 |
|---------|-------------------|
| ロック済みシフトの保持 | メイン枠に常勤医がいる日のサブ枠を「不要」としてロック |
| 外部医師の除外 | サブ枠対象タグの医師だけを渡す（例: 消化器内科だけ） |
| 制約設定の差し替え | サブ枠用の制約セット（間隔0・連続OK等）をまるごと渡す |
| 未ロック枠だけ再生成 | 空いているサブ枠だけ埋める |

**処理の流れ:**
1. メイン枠を通常通り自動生成（1回目のソルバー実行）
2. メイン枠の結果を見て、サブ枠の状態を自動判定:
   - 常勤医が入った日 → サブ枠をロック（不要）
   - 外部枠の日 → サブ枠を空欄（生成対象）
3. ソルバーを2回目実行:
   - 対象医師: サブ枠対象タグの医師のみ（属性でフィルタ）
   - 制約セット: サブ枠用（連続OK、間隔0日、月上限なし等）
   - ロック済みの日は飛ばして、空いている日だけ埋める
4. 結果をサブ枠（枠番号=2）として保存

**ポイント:**
- ソルバー自体の改修は不要。入力データ（医師リスト・制約・ロック情報）を組み替えるだけ
- APIに「サブ枠生成」エンドポイントを1つ追加する程度
- UIは「サブ枠を生成」ボタンを追加（メイン枠の生成後に押す2段階操作）
- **工数**: 1〜2週

#### Step 3: 同時最適化 ★★★★★

メイン枠とサブ枠を一つの最適化モデルで同時に解く。需要が出ない限り着手しない。
Step 2（ソルバー2回実行）で実用上十分なため、優先度は極めて低い。

### [B] 属性別制約（科ごとのルール）

- **ニーズ**: 複数科が合同で1つの当直表を作る場合、科ごとに異なるルールがある。例:

  - 消化器科は月3回以上（科別の最低回数）
- **現状の限界**: 全制約は全医師に一律適用。科ごとに変えられない
- **前提**: [A] 属性システム

- **枠番号との関係**: 属性別制約はメイン枠にもサブ枠にも独立して適用可能。
  例: メイン枠では「泌尿器科は間隔5日」、サブ枠では「専攻医は制約なし」
  枠ごとに属性別制約を設定する形にすれば、制約の交差なく自然に組み合わさる

## 軸③ シフト種別

### なぜ軸③だけ本質的に難しいのか

軸①（枠番号）と軸②（属性）は既存の仕組みの延長で実現できた。
軸③が根本的に難しいのは、**複数のシフト種別に同じ医師が入り、種別をまたいで間隔制約を共有する**から。

**順番生成（ソルバー2回実行）が通用しない例:**

```
1回目: 救急当直を生成
  → 4/1 山田、4/5 山田

2回目: ICU当直を生成（救急の結果を参照）
  → 4/3 に山田を入れたいが、4/1(救急)と4/5(救急)の間で間隔違反
  → 解なし or 非効率な配置になる
```

1回目の生成時点では2回目の結果がわからないので、全体最適にならない。
**全種別の変数を1つのモデルに入れて同時に解くしかない** → 自動生成エンジンの全面書き換え。

**枠番号のソルバー2回実行ではなぜ問題にならなかったか:**

| パターン | 種別間で間隔を共有するか | 順番生成 |
|---------|----------------------|---------|
| メイン当直 + オンコール（サブ枠） | しない（オンコールは連続OK） | **できる** |
| メイン当直 + 専攻医（サブ枠） | しない（同日配置なので間隔問題なし） | **できる** |
| 救急当直 + ICU当直 | **する**（同じ医師が別日に入り間隔が必要） | **できない** |

これはいわゆる**ナーススケジューリング問題**（NPS: Nurse Scheduling Problem）の本質。
3交代制（日勤/準夜/深夜）も、救急科の複数ポジション同時当直も、「複数種別×同一人物×間隔制約」という同じ構造を持つ。学術的にもNP困難と知られている組合せ最適化問題であり、CP-SATで解けはするが、変数と制約の設計に専門知識が必要になる。

### [D] カスタムシフト種別（軸③の基盤）

- **ニーズ**: 「ICU当直」「外来当直」「救急当直」など病院独自のシフト種別。種別ごとにスコアが異なる
- **現状の限界**: 日直/当直の2種類がコード内に固定。追加できない
- **想定されるシステム要件**:
  - 「シフト種別」テーブル新設（施設ごとに名前・スコア・適用曜日・時間帯を定義）
  - 自動生成エンジン: 変数を「医師×日×シフト種別」の3次元に拡張。**全面的な書き換え**
    - 全種別を1つのモデルで同時に解く必要がある（順番生成では種別間の間隔制約を満たせない）
    - 変数数: 現在の「医師×日×2種」→「医師×日×N種」に増加
    - 制約: 種別をまたいだ間隔制約・スコア合算・不可日チェックすべてが種別横断になる
  - 当直記録テーブル: シフト種別の文字列 → シフト種別テーブルへの参照に変更
  - 画面: スケジュール表の列数が可変に。D&D・パレット・出力すべてに波及
- **枠番号との関係**: 枠番号 × シフト種別で「メイン枠の当直」「サブ枠のICU当直」のような組み合わせも可能。ただし複雑さが掛け算になるので、最初は枠番号を先に作り、シフト種別は後から追加する方が安全
- **難易度**: ★★★★☆（全レイヤーに波及する最大級の改修。ナーススケジューリング問題と同じ構造）
- **追記日**: 2026-03-28

### [J] 条件付きシフト — 当直/オンコール排他生成

- **ニーズ**: 同一科に「当直」と「オンコール」がある。当直がいる日はオンコール不要、いない日にオンコールを立てる。オンコールは連続勤務OKなど別ルール
- **前提（本格実装）**: [D] カスタムシフト種別 + [B] 属性別制約
- **枠番号+ソルバー2回実行での実現（仮実装）**:
  軸③の本格実装を待たずに、枠番号方式の Step 2（ソルバー2回実行）で実現可能:
  1. メイン枠を自動生成（1回目）→ 常勤医が入った日と外部枠の日が確定
  2. メイン枠に常勤医がいる日のサブ枠をロック（オンコール不要）
  3. 外部枠の日のサブ枠だけを対象に、サブ枠用の制約（連続OK・間隔なし）+ 対象医師（消化器内科タグ等）でソルバーを2回目実行
  4. 結果をサブ枠（枠番号=2）として保存
  排他関係（当直あり→オンコール不要）は「メイン枠の結果を見てサブ枠をロック」で自然に表現できる
- **難易度**: ★★★★☆（軸③の本格実装） / **枠番号での実現なら ★★★☆☆**（[E] Step 2 + [A] 属性があればほぼそのまま使える）
- **追記日**: 2026-03-28

## 独立要件（どの軸とも無関係）

### [H] 複数月一括最適化

- **ニーズ**: 3ヶ月を一括で組みたい

### 各切れ目で何ができるようになるか

| 切れ目 | できること | できないこと |
|--------|-----------|------------|
| **①** 枠番号のみ | 2人体制の手動D&D配置。専攻医をサブ枠に。サブ枠独自の簡易ルール | 科別ルール。サブ枠の自動生成 |
| **②** 枠+属性 | 科ごとの間隔・回数・曜日制限。科バランス。「専攻医だけサブ枠にドロップ可」 | サブ枠の自動生成。シフト種別追加 |
| **③** 枠×属性+自動 | 階層ペアリング完成。**サブ枠の自動生成（ソルバー2回実行）**。オンコールの自動配置 | シフト種別の正式な追加 |
| **④** 3軸全部 | カスタムシフト種別。排他ルールの正式対応 | ナース3交代。スキルミックス |

### なぜこの順番か

- **枠番号を最初にやる理由**: 専攻医の2人体制は具体的なニーズがある。かつ枠番号の追加は既存のデータ構造への影響が小さい（列1つ追加）
- **属性を枠の後にやる理由**: 属性だけあっても制約に反映しないと意味がない。枠番号が先にあれば「サブ枠に専攻医だけ」のような属性の活用先がすでにある
- **サブ枠自動生成が切れ目③で実現できる理由**: 既存ソルバーを制約・医師リストを差し替えて2回目実行するだけ。新しい仕組みは不要。メイン枠の結果からサブ枠の対象日を自動判定し、ロックで制御する。オンコールの排他（当直あり→オンコール不要）もこの仕組みで自然に表現できる
- **シフト種別を最後にする理由**: 自動生成エンジンの全面改修が必要で最も重い。枠番号+属性+ソルバー2回実行でオンコール等は十分実現できるので急がない
- **散らからないためのポイント**: 各軸が独立しているので、「枠番号だけ入れて属性はまだ」「属性を入れたけどシフト種別はまだ」という中間状態でもシステムは整合的に動く。後から足しても既存を壊さない

## マルチ製品戦略 — 当直版・ナー���版・救急版の共存

### 基本方針: 1コードベース・1デプロイ・製品モードで切り替え

コードベースを分岐（別リポ・別ブランチ）させない。
施設テーブルに「製品モード」を持ち、同じアプリ内でモードに応じた画面と生成エンジンを出し分ける。

**理由:** 個人開発でコードを分岐させるとバグ修正・セキュリティ対応・共通機能の改善を複数箇所に当てる運用が破綻する。

### 共通基盤 vs 製品別モジュール

**バックエンド:**

| 共通基盤（全製品で1つ） | 製品別（モードごとに差し替え） |
|---|---|
| 認証・マルチテナント | 生成エンジン（optimizer実装） |
| スタッフ管理（名前・属性・不可日） | シフト種別のデフォルト定義 |
| 枠番号システム | 制約のデフォルト値 |
| 属性タグシステム | スコア計算ルール |
| スケジュール保存・仮保存・確定 | 事前チェック・解なし診断 |
| 設定KVストア | |
| マジックリンク・共有入力 | |

**フロントエンド:**

| 共通（そのまま使える） | 製品ごとに新規作成 |
|---|---|
| ログイン・登録・認証 | スケジュール表（列構造が根本的に違う） |
| D&Dの仕組み（ドラッグ検知・ロック・Undo） | 制約設定画面（設定項目が違う） |
| ウィザードの枠組み | ウィザードの質問内容 |
| パレット・マジックリンク | レポートのグラフ・集計ロジック |
| アカウント管理 | PDF/Excel出力（列構造が違う） |

**フロントは「骨格は共通だけど、画面は作り直し」。** 3交代×10人のナース表と2列の当直表は見た目も操作感もまったく別物。

### なぜ製品が分かれるのか — 本質的な違い

| | 当直版（現在） | ナース版 | 救急版 |
|---|---|---|---|
| シフト種別 | 日直/当直（2種） | 日勤/準夜/深夜（3種） | 救急/ICU/外来（N種） |
| 1日あたり人数 | 各1人 | 各3〜10人 | 各1〜3人 |
| **種別をまたいだ間隔** | **不要** | **必要**（深夜→翌日勤禁止） | **必要** |
| 規模 | 5〜20人 | 20〜100人 | 10〜30人 |

**種別をまたいだ間隔制約** がナース版・���急版の本質的な難しさ。
全種別を1つのモデルで同時に解くしかない（順番生成では解けない）。
これはナーススケジューリング問題（NPS）として学術的にも知られた困難な問題。
当直版では不要だったこの制約が、生成エンジンの設計を根本から変える。

### [D] カスタムシフト種別 = マルチ製品化の分岐点

[D]を実装するタイミングで、自然にマルチ製品化の基盤が整う:

1. シフト種別をDBテーブル化（コードのハードコードを撤廃）
2. 制約定義もDBテーブル化（種別ごとの間隔・上限・スコア）
3. 生成エンジンを「共通の枠組み」と「製品別の実装」に分離

```
分離後の構造:
  optimizer_base.py    ← 共通: 変数生成・スコア計算・診断の枠組み
  optimizer_doctor.py  ← 当直版: 2種、種別またぎなし（現在のロジック）
  optimizer_nurse.py   ← ナース版: 3交代、種別またぎ間隔あり、ローテーション
  optimizer_ed.py      ← 救急版: N種同時、種別またぎ間隔あり
```

**[D]を実装するまでは製品分離を考える必要がない。** 枠番号+属性は当直版の中だけで進められる。

### 進化パスのまとめ

```
今〜切れ��③    当直版に集中。枠番号+属性。製品分離は不要
    ↓
切れ目④ [D]    シフト種別DB化。生成エンジンを共通部+製品別に分離
    ↓           施設テーブルに製品モード追加
    ↓           ← ここが「1製品→マルチ製品」の分岐点
    ↓
ナース版        optimizer_nurse.py を実装 + フロント画面を新規作成
                新規登録時に「ナース版」を選ぶだけで使える
    ↓
救急版          optimizer_ed.py を実装 + フロント画面を新規作成
```

## 要件の詳細（個別の背景・具体例）

> 以下は各要件の詳細な背景情報。設計判断の根拠として残す。

### [K] 階層型ペアリングの背景

**現場の運用例:**
- 上級医・病棟医・専攻医の3職位がある
- 専攻医は一人で当直できない → 必ず上級医とペア（ハード制約）
- 病棟医は一人でもOK
- 上級医は一人でも可だが、できれば専攻医をつけたい（ソフト制約）

**なぜ「ただ2人にする」では足りないか:**
- 「誰と誰を組ませるか」に属性の依存関係がある
- 病棟医がメインの日に専攻医をサブにつけてはいけない
- この判定はD&Dのドロップ可否で表現するのが最もシンプル

**枠番号+属性方式での実現:**
- 専攻医を「サブ枠専用」で登録
- D&D時: サブ枠にドロップ → その日のメイン枠の医師の職位タグを確認 → 上級医ならOK、病棟医ならNG
- 自動生成エンジンには一切影響しない

**2枠目の自動配分は現実的でない理由:**
メインに誰を配置するかによってサブの可否が変わるため、メインとサブを同時に最適化する必要がある。変数・制約が掛け算で増え、実装も保守も困難。実運用でも専攻医の配置は上級医や科長が手動で決めることが多く、自動化のニーズは低い。

### [J] オンコールの背景

**現場の運用例（消化器内科）:**
- 当直医がいる日: 当直のみ（オンコール不要）
- 当直医がいない日: オンコール体制（電話対応のみ、連続OK）
- オンコールのルールは当直と全く異なる（連続可、間隔なし、負担スコア低い）

**枠番号での仮実装:**
- メイン枠 = 当直（通常ルール）
- サブ枠 = オンコール（独自ルール: 連続OK、間隔0日）
- 「当直がいない日にオンコール」の排他関係は、管理者がD&Dで手動設定
- 本格的な排他自動生成は軸③（シフト種別）の実装後

### [B] 科別ルールの具体例

| 診療科 | ルール例 |
|--------|---------|
| 泌尿器科 | 間隔5日（厳しめ）。土曜は月1回だけ |
| 整形外科 | 平日のみ担当（土日祝は対象外） |
| 消化器科 | 月3回以上（最低回数あり） |
| 外科 | 全日程OK・標準ルール |

**属性別制約の適用イメージ:**
- 全体ルール: 間隔4日、土曜月1回
- 泌尿器科の上書き: 間隔5日、土曜月1回（変更なし）
- 整形外科の上書き: 平日のみ（曜日制限追加）
- 消化器科の上書き: 最低3回（最低回数追加）
- 上書きがないタグの医師には全体ルールが適用される
