## フェーズ1b：ランディングページ（`/`）

### 目的
- 初見ユーザーが「何のアプリか」を即座に理解できる
- ログイン・新規登録・デモ体験への導線を提供する

### コンテンツ構成

```
■ ヘッダー（固定）
  [シフらく ロゴ]                    [ログイン] [無料で始める]

■ ヒーロー
  「当直表づくり、まだExcelですか？」
  条件を入れるだけで、公平な当直表を自動作成。
  [無料で始める] [デモを試す]

■ ペインポイント（3カラム）
  1. 「誰かに偏ってしまう」→ スコアで公平性を自動管理
  2. 「希望日の調整が大変」→ 医師が自分で不可日を入力（マジックリンク）
  3. 「作り直しが面倒」→ ドラッグ&ドロップで即修正

■ 3ステップで使える
  1. 人数とルールを入力
  2. 自動生成ボタンを押す
  3. 必要なら手動で微調整
  （各ステップにスクリーンショットまたはアニメーション）

■ インラインデモ（フェーズ5で実装、それまではCTAのみ）
  「今すぐ試してみる」→ 人数と基本ルールだけ入れて即生成

■ 登録CTA（ページ最下部）
  「無料で使ってみる」→ /register へ
```

## フェーズ2a：共通フック整理

### 目的
`/app`と`/dashboard`でロジックを二重管理しないよう、共通の状態管理層を切り出す。

### 設計

```
src/app/hooks/
  useOnCallCore.ts       ← 【新規】統合フック
                            既存の useDashboardState + useScheduleApi +
                            useScheduleDnd + useScheduleHistory +
                            useRealtimeScores + useOptimizerConfig +
                            useCustomHolidays + useHolidays を束ねる

src/app/dashboard/page.tsx  ← useOnCallCore() を使う（既存UIそのまま）
src/app/app/page.tsx        ← useOnCallCore() を使う（新UI）
```

### 段階的リファクタ
1. まず現行page.tsxのフック呼び出し部分をそのまま`useOnCallCore`に移動
2. `/dashboard`が動くことを確認
3. その後`/app`で同じフックを使う

## フェーズ3：セットアップウィザード

### 発動条件
初回ログイン時（`system_settings`に`setup_completed`キーが存在しない場合）

### 質問フロー（技術用語を使わず日常の言葉で）

```
Step 1: 「当直の形式を教えてください」
  → 日直と当直は別の人ですか？
  → [別々の人が担当] [同じ人が両方担当]
  → holiday_shift_mode を設定

Step 2: 「何人で当直を回していますか？」
  → 数値入力
  → 入力した人数分の仮医師（「医師1」「医師2」...）を自動生成

Step 3: 「1ヶ月に1人あたり何回くらい当直しますか？」
  → 「最低○回〜最大○回」のスライダー
  → 裏でスコア換算して min_score / max_score を設定
  → ユーザーには「回数」として見せる

Step 4: 「当直明けは何日空けたいですか？」
  → [翌日OK] [1日] [2日] [3日] ボタン選択
  → interval_days を設定

Step 5: 「土曜の夜間当直は月何回まで？」
  → [制限なし] [1回] [2回] ボタン選択
  → max_saturday_nights を設定

Step 6: 完了
  → 「設定完了！サンプルのシフトを生成してみましょう」
  → 自動で生成実行 → 結果を見せる
  → 「この設定はあとからいつでも変更できます」
```

### 技術的な注意
- ウィザードの各ステップは1コンポーネント（`SetupWizard.tsx`）で管理
- 完了時に`system_settings`へ`setup_completed: true`を保存
- 仮医師生成は`POST /api/doctors/`を人数分呼び出し

## フェーズ4：オンボーディング

### 管理方法
- `system_settings`に`onboarding_seen`キーで配列を保存（DB永続化）
- 例: `["rules", "weights", "doctors", "generate", "dnd"]`
- 「チュートリアルをもう一度見る」ボタンを設定メニュー内に設置

### 各セクションの説明内容

| セクション | トリガー | 説明ポイント |
|-----------|---------|------------|
| 基本ルール | ドロワー初回オープン | 「守らなければならないルールです。当直間隔と上限回数から設定しましょう」 |
| 優先度の調整 | ドロワー初回オープン | 「できれば守りたいルールの強さです。数値が大きいほど強く考慮されます」 |
| 医師の管理 | ドロワー初回オープン | 「医師の名前・目標回数・お休みの日を設定できます」 |
| スケジュール生成 | 初回生成完了時 | 「設定に基づいて最適なシフトが作られました。色分けの見方を説明します」 |
| D&D編集 | 初回生成完了時 | 「ドラッグ&ドロップで手動調整できます。ロックすれば固定されます」 |

## フェーズ5：公開デモ

### バックエンド

```
POST /api/demo/optimize
  - 認証不要
  - レート制限: 1分3回
  - hospital_id 不要（リクエストボディだけで完結）
  - DB 書き込みなし
  - 医師数上限: 15人
  - レスポンスは通常の OptimizeResponse と同じ形式
```

### フロントエンド
- LPに**インラインデモ**として埋め込み（別ページ不要）
- 入力: 医師人数（スライダー）+ 基本ルール（当直間隔・土曜上限）
- 出力: 生成されたスケジュール表（閲覧のみ、D&D不可、保存不可）
- 常時CTA: 「本番で使うには無料登録 →」
