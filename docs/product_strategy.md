# シフらく プロダクト戦略・ロードマップ

> 2026-03-20 策定

## Context

3週間で個人開発したシフらく（oncall-app）を、「売れるプロダクト」にするための包括的な戦略。
技術的には十分な土台がある。問題は「何を作るか」「何を作らないか」「どう売るか」。

---

## 1. 競争優位性（なぜ勝てるのか）

| 強み | 詳細 |
|------|------|
| **数理最適化（CP-SAT）** | 競合はExcel手作業かルールベース。シフらくはOR-Toolsで**数学的に公平なスケジュール**を生成する。これは模倣困難 |
| **マジックリンク** | 医師が個別アカウント不要で不可日入力。LINEでURL送るだけ。この手軽さは競合にない |
| **ドメイン知識** | 開発者自身が病院勤務。combined/splitモード、土曜1.5倍、間隔制約など**現場を知っている設計** |
| **開発速度** | ソロ開発者の反復速度はエンタープライズ競合の10倍。顧客フィードバック→実装が1日で可能 |

**一言で言うと：** 「Excelより圧倒的に楽で、Dr.JOYより圧倒的に軽い」ポジション

---

## 2. 致命的な弱点（優先度順）

### Tier1: これがないと売れない（採用阻害）

| # | 弱点 | なぜ致命的か |
|---|------|------------|
| **W1** | PDF/Excel出力がない | 当直表は**壁に貼る紙**。出力できないツールは完成品ではない |
| **W2** | 「解なし」が不親切 | 制約が矛盾したとき「No feasible」だけ。非IT医師は「壊れた」と思う |
| **W3** | 通知機能がない | マジックリンクを手動でLINEコピペ。スケジュール確定の通知もない |
| **W4** | プライバシーポリシー/利用規約がない | 稟議書に添付する書類がゼロ。法的にもNG |
| **W5** | データ削除/エクスポートがない | 「やめたいとき全データ消せますか」に答えられない |

### Tier2: これがないと3ヶ月で離脱する（継続阻害）

| # | 弱点 | 影響 |
|---|------|------|
| **W6** | 不可日が0/1のみ | 「できれば避けたい」を表現できない（is_soft_penaltyフィールドはあるがUI未公開） |
| **W7** | 月単位のみ | 2〜3ヶ月先を一括で組みたい病院がある |
| **W8** | シフト2種類のみ | ICU当直・外来当直など部門別シフトに対応できない |
| **W9** | 分析/レポートがない | 半年使って「この1年公平だった？」が見えない |

## 3. ロードマップ

### Phase 1: 売り物にする（1〜3ヶ月目）

**テーマ：「自分の病院以外にも使える状態にする」**

#### Month 1: 最低限の完成品

| 優先度 | 施策 | 工数 | 理由 |
|--------|------|------|------|
| **P0** | PDF/Excel出力 | 3-5日 | バックエンドに`reportlab`/`openpyxl`追加。`GET /api/schedule/{year}/{month}/export?format=pdf` |
| **P0** | 制約矛盾の診断 | 5-7日 | CP-SATの`AddAssumptions`で制約を1つずつ緩和→「田中先生と鈴木先生の不可日が4/15（日曜）で競合」のようなメッセージ |
| **P0** | プライバシーポリシー・利用規約 | 1-2日 | `/privacy`, `/terms` 静的ページ。テンプレート利用でOK |
| **P1** | 不可日ソフト化UI | 2-3日 | `/entry/[token]`に「できれば避けたい / 絶対無理」トグル。DBフィールドは既存 |
| **P1** | マジックリンク一斉送信 | 3-5日 | メール or LINE Messaging APIで全医師にURLを自動送付 |

#### Month 2: 実証

| 優先度 | 施策 | 工数 | 理由 |
|--------|------|------|------|
| **P0** | **自院で2ヶ月以上の実運用** | 継続 | これが最重要。「月N時間削減」の定量データを取る |
| **P1** | 簡易分析ページ | 3-5日 | 過去6ヶ月のスコア分布を医師別棒グラフで表示 |
| **P1** | 操作ログ | 2-3日 | 誰が生成・確定・編集したかを`system_settings`にJSONBで記録 |
| **P2** | PWA化 | 1日 | `next-pwa`でmanifest追加。ホーム画面追加でアプリ風に |

#### Month 3: 課金開始

| 優先度 | 施策 | 工数 | 理由 |
|--------|------|------|------|
| **P0** | Stripe決済 | 3-5日 | Stripe Checkout。Webhookでプラン管理 |
| **P0** | プラン別機能制限 | 2-3日 | `hospitals`テーブルに`plan`カラム追加 |
| **P1** | データエクスポート/アカウント削除 | 2日 | GDPR的な最低限。JSON全データダウンロード + 退会機能 |

### Phase 2: 定着させる（3〜6ヶ月目）

**テーマ：「Excelに戻れなくする」**

| 施策 | 工数 | 理由 |
|------|------|------|
| カスタムシフト種別 + 日別シフト属性 | 3-4週 | `shift_types`テーブル新設。日にタグをつけてタグ別スコア・医師別上限（広域当番など）。optimizerをシフト種別非依存に |
| 2人体制スロット | 2-3週 | 日別required_count。AddExactlyOne→動的人数。スケジュール表の複数医師表示対応。ニーズは救急科・大規模病院に限定的 |
| LINE連携 | 1-2週 | 確定通知をLINE公式アカウントでプッシュ。日本の病院はLINEが連絡手段 |
| Googleカレンダー同期 | 3-5日 | ICSフィード（`/api/schedule/ical/{doctor_token}`）。OAuth不要 |
| シフト交換リクエスト | 2週 | 確定後に医師AがBにスワップ申請→制約チェック→承認。現場の最重要ニーズ |
| 複数月一括最適化 | 2-3週 | 逐次最適化（carry-forward）方式で。CP-SATモデルサイズ爆発を回避 |

### Phase 3: 拡大する（6〜12ヶ月目）

**テーマ：「日本の当直表のデファクトになる」**

| 施策 | 理由 |
|------|------|
| 複数診療科対応 | 1病院内で内科・外科・救急が別スケジュール。`department_id`追加 |
| HIS連携API | 電子カルテベンダー向けAPI公開。医師マスタの自動同期 |
| ISMS/セキュリティ文書 | 中規模病院の稟議を通すため。ISO 27001は不要だが、セキュリティホワイトペーパーは必要 |
| テンプレート/プリセット | 「繁忙期テンプレ」「夏休みテンプレ」の保存・読込。`weight_presets`テーブルは既存 |

## 4. 価格戦略

| プラン | 月額（税込） | 対象 | 制限 |
|--------|-------------|------|------|
| **無料** | 0円 | お試し・小規模 | 医師8名まで・全機能使える・**出力にウォーターマーク**・履歴1ヶ月 |
| **スタンダード** | 3,000円 | 5〜20名の診療科 | 医師25名・PDF/Excel出力・マジックリンク・12ヶ月履歴・メール通知 |
| **プロ** | 5,000円 | 大規模・複数科 | 医師50名・複数科・LINE連携・分析・API・優先サポート |

**なぜこの価格か：**
- 3,000円/月は**稟議不要の経費精算ライン**（多くの病院で診療科長の裁量で通る）
- 当直表担当医の時給1万円超 × 月4-8時間 = 月4-8万円の労働を代替。ROI 10倍以上
- **無料プランの制限はPDF出力**。生成・編集は全機能使える→「便利だ」と実感させてから課金

**やってはいけないこと：**
- 従量課金（医師数×N円）→ 病院の予算管理に合わない
- 年契約の強制 → 初期は月額で始めて、定着後に年割（2ヶ月無料）を提案

## 5. Go-to-Market（最初の10病院の獲得方法）

### Step 1: 自院実証（Month 1-2）
- 自分の診療科で2ヶ月以上使う
- 定量データを記録：「月5時間→30分に短縮」「不公平クレームゼロ」

### Step 2: 人脈営業（Month 2-4）
- 知り合いの医師3-5人に「無料で使ってみて」
- **勉強会/症例検討会で5分デモ**。医師は同僚の推薦を信頼する
- 医局のLINEグループにデモURL共有

### Step 3: コンテンツ（Month 3-6）
- Qiita/Zenn: 「CP-SATで当直表を自動化した話」（技術記事→開発者の認知）
- note: 「当直表作りを30分で終わらせる方法」（非技術者向け→医師の認知）
- YouTube: 3分デモ動画（登録→生成→調整→確定の全フロー）
- SEO: 「当直表 自動作成」「当直 シフト管理 アプリ」

### Step 4: パートナーシップ（Month 6+）
- 医業経営コンサルタントへのアプローチ
- 都道府県医師会の勉強会での発表

**やっても無駄なこと：**
- 病院IT部門への飛び込み営業（門前払い）
- HOSPEX等の展示会出展（コスト見合わない）
- 営業チーム採用（時期尚早）

## 6. スピンオフ製品

### やるべき: 「ドクター不可日収集ツール」（無料スタンドアロン）

マジックリンクの不可日収集機能を**単体ツール**として切り出す。
- Google FormsやLINEで不可日を集めている病院が対象
- 無料で使える → 「自動生成もしたいな」→ シフらくへのアップセルファネル
- 実装コスト低（既存の`/entry/[token]`をほぼそのまま使える）

### やらないこと

| 候補 | やらない理由 |
|------|------------|
| スケジューリングAPI販売 | HISベンダー提携が前提。今は需要なし |
| 汎用シフト管理ツール | 競合多すぎ。当直特化が強み |
| 病院コミュニケーションツール | Dr.JOYの領域。勝てない |

---

## 7. 作ってはいけないもの

| 施策 | なぜダメか |
|------|----------|
| ネイティブアプリ（iOS/Android） | PWAで十分。2プラットフォーム維持はソロ開発者を殺す |
| リアルタイム共同編集 | 当直表は1人が作る。WebSocket複雑性に見合わない |
| AI自然言語入力 | 「AIに制約を話しかける」は見栄えだけ。構造化UIの方が正確 |
| 多言語対応 | 市場は日本の病院。今やるのはリソースの無駄 |
| BI/ダッシュボードビルダー | 簡易分析ページで十分。病院は独自BIを持っている |
| 3交代制のハードコーディング | カスタムシフト種別の汎用システムを先に作る。個別対応は技術的負債 |

## 8. 技術的な準備事項

| 項目 | 対応 |
|------|------|
| optimizer.pyのリファクタ | カスタムシフト種別の前にmixin分割（Task 2.6-5）を完了させる |
| テスト整備 | pytest修復。最低限：optimizer単体テスト + API統合テスト |
| CORS本番設定 | `FRONTEND_URL="*"` → 本番ドメイン限定 |
| DB接続プーリング | Neonのコネクションプーラー明示指定 |

## 9. 1年後のゴール

- **有料病院30〜50施設**
- **MRR 10〜25万円**（3,000〜5,000円 × 30〜50）
- カスタムシフト対応済み・複数科対応済み
- 「日本の中小病院の当直表 = シフらく」というポジション確立

**最大のリスクは技術ではない。病院の導入スピードの遅さとソロ開発者の燃え尽き。**
優先順位を絞り、作らないものを明確にし、最小限で最大の効果を出すことが生命線。
