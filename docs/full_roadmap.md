### P2-2: AI制約エンジニアリング ★★★☆☆

**ニーズ：** AIガイドがアドバイスだけでなく、実際にAPIを叩いて設定を変更する

**Phase1のAIガイドとの違い：**
| Phase1（P1-20） | Phase2（P2-2） |
|-----------------|---------------|
| 「この設定をこう変えてください」とアドバイス | 「この設定を変更しますか？」→ 承認 → AIが実行 |
| ユーザーが手動で設定変更 | AIがAPIを叩いて設定変更 |
| 知識ベース参照のみ | 知識ベース + API操作権限 |

**実装：**
| # | 内容 |
|---|------|
| 2-2a | 既存のハード制約・重みパラメータにAI操作可能フラグを追加（optimizer_configのJSON内にスロット） |
| 2-2b | AIガイドにfunction calling / tool use機能を追加 — 設定読み取り・変更のAPIを呼べるように |
| 2-2c | **必ず確認画面を挟む** — AIが「こう変更しますか？」→ ユーザーが承認 → 実行。AIが勝手にやらない |
| 2-2d | 変更履歴の記録（usage_eventsに `ai_setting_change` イベント） |

**例：**
> ユーザー「前月日祝勤務した医師は翌月日祝なしにしたい」
> AI「わかりました。前月のシフトデータを確認して、該当する医師の今月の日祝日に不可日を自動設定するルールを追加しますか？」
> ユーザー「お願い」
> AI（前月シフト参照 → 該当医師特定 → 不可日API叩く → 完了報告）

**product_strategy.md 修正点：**
「AI自然言語入力は作ってはいけない」→ **UI代替としてのAI入力はやらない。構造化UIの上にAIが操作レイヤーとして乗る形でやる。**

**目的：** 不活化ウェイト削除 + 新アルゴリズム実装 + mixin分割

- 非公開ハード設定: `prevent_sunhol_consecutive`, `respect_unavailable_days`, `max_sunhol_days`, `max_sunhol_works`

**新規実装：**
| # | 内容 | 置き換え対象 |
|---|------|------------|
| 3a | 加重累積均等化 | sunhol_fairness + sat_month_fairness + past_*_gap + sat_consec + *_3rd |
| 3b | 累積目標乖離 | score_balance + month_fairness → 符号付き累積で長期公平性 |
| 3c | optimizer.py mixin分割 | 1ファイル→制約/変数/目的関数を分離 |

**新UIイメージ：**
```
勤務間隔（ハード）    [4日]
理想間隔              [7日] ← ソフト自動追従
土曜当直上限          [1回]
土日祝合算上限        [3回]

日祝シフトモード      [別々/日当直]
```

上級設定の重みスライダーは原則不要になる（ハードに追従してソフトが自動生成）。

**実装（future_schedule_requirements.md [E] Step 2）：**
1. メイン枠を通常通り自動生成（1回目のソルバー実行）
2. メイン枠の結果を見て、サブ枠の状態を自動判定
3. ソルバーを2回目実行（対象医師・制約を差し替え）
4. 結果をサブ枠として保存

**ポイント：** ソルバー自体の改修不要。入力データの組み替えだけ。

**前提：** P2-4（枠番号Step1）

### P3-3: オンコール — 枠番号+自動生成で実現 ★★★☆☆

**ニーズ：** 当直あり→オンコール不要、なし→オンコール配置

**実装（future_schedule_requirements.md [J] 仮実装）：**
- P3-2の応用。メイン枠に常勤医がいる日のサブ枠をロック → 残りを対象科でソルバー2回目実行
- 排他関係（当直あり→オンコール不要）は「メイン枠の結果を見てサブ枠をロック」で自然に表現

**前提：** P3-2（サブ枠自動生成）+ P2-5（属性でフィルタ）

### P3-4: カスタムシフト種別基盤 ★★★★☆

**ニーズ：** ICU当直・救急当直・オンコール等の独自シフト種別

**これが最大の改修。optimizer全面書き換え。**

**実装（future_schedule_requirements.md [D]）：**
| # | 内容 |
|---|------|
| 4a | `shift_types` テーブル新設（hospital_id, name, score, applicable_weekdays, time_range） |
| 4b | shift_assignmentsのshift_type列をshift_types参照に変更 |
| 4c | optimizer: 変数を「医師×日×シフト種別」の3次元に拡張 |
| 4d | 全種別を1つのモデルで同時に解く（種別間の間隔制約を満たすため） |
| 4e | D&D画面の列数が可変に。パレット・出力すべてに波及 |

**なぜ難しいか：** 複数シフト種別で同じ医師が別日に入り、種別をまたいで間隔制約を共有するため、順番生成では全体最適にならない。全種別の変数を1モデルで同時に解く必要がある。

**前提：** P2-3（V3 Optimizer）完了推奨
