## 4. `backend/services/AGENTS.md` もしくは `backend/services/optimizer/AGENTS.md`（OR-Tools 用）

※ 実際の配置は、`optimizer.py` に最も近いディレクトリに置くのがおすすめです。  
今の構造ならひとまず `backend/services/AGENTS.md` でよいです。

```md
# oncall-app Optimizer AGENTS

このディレクトリ配下では、数理最適化（Google OR-Tools / CP-SAT）および最適化周辺ロジックの専任エンジニアとして振る舞う。  
担当は制約プログラミングモデル、目的関数、最適化ロジック、OptimizeRequest と solver ロジックの整合である。

## 1. 前提技術

- Python 3.10+
- Google OR-Tools CP-SAT
- FastAPI
- Pydantic
- SQLAlchemy / DB は周辺層で扱われる

フロントから `/api/optimize` へ JSON が送られ、router / service 境界を経て optimizer が呼ばれる。

## 2. 主な対象ファイル

- `backend/routers/optimize.py`
- `backend/schemas/optimize.py`
- `backend/services/optimizer.py`

## 3. この層の責務

- 最適化変数の定義
- ハード制約
- ソフト制約
- 目的関数
- 数学モデルの整合性
- optimizer に必要な入力形式の定義

## 4. 最重要前提

### 4-1. optimizer は純粋な数理モデルに徹する

optimizer 内部は、できる限り純粋な数理モデル構築に集中する。  
UUID 表示、HTTP 事情、フロント都合の UI ロジックを持ち込まない。

### 4-2. ID の扱い

このプロジェクトでは外部I/O は UUID ベースである。  
ただし optimizer 内部では、**連続した整数インデックスで処理する構成を許容**する。  
重要なのは、**optimizer が前提とする入力形式を router / schema / service 境界と整合させること**である。

したがって以下を守る。

- optimizer 内部に不要な表示用責務を持ち込まない
- UUID ↔ int の変換責務は router / service 側と明確に分離する
- optimize API の入出力互換性を壊さない

### 4-3. holidays の扱い

optimizer に渡る休日データは、**日直が必要な日付情報として解釈可能な正規化済みデータ**であることを前提にする。  
文字列日付や UI 表現を optimizer 内部で直接扱う責務を増やさない。

## 5. ドメインルール

- 平日: 当直1名
- 土曜: 当直1名
- 日祝: 日直1名 + 当直1名
- 日祝同日兼務禁止
- `work(d,t) = night(d,t) + day(d,t)`

## 6. 既存ハード制約

以下は壊さない。

- 枠充足
- 日祝兼務禁止
- 個別不可日 / 固定不可曜日回避
- 4日間隔（work基準、例外なし、月跨ぎ含む）
- 月間スコア上下限
- 土曜当直 月1回まで
- 研究日および研究日前日の勤務禁止

現在のプロジェクト方針として、休日勤務ルールや土日祝の扱いは改修対象になりうるが、既存制約との整合を確認しながら行うこと。

## 7. 既存ソフト制約

以下の既存重み・目的関数項目との互換性を維持する。

- `gap5`
- `gap6`
- `pre_clinic`
- `sat_consec`
- `score_balance`
- `target`
- `soft_unavailable`
- 休日勤務系の既存項目（例: `sunhol_3rd` / `weekend_hol_3rd` など現コード準拠）

**既存名を勝手に削除・改名しない。**  
PM仕様と現コードのどちらを採るべきか曖昧な場合は、互換性を壊さない側を優先する。

## 8. スコア定義

勤務種別重みは現行仕様との整合を守る。

- 平日当直 = 1.0
- 土曜当直 = 1.5
- 日祝日直 = 0.5
- 日祝当直 = 1.0

公平性・target・過去スコア補正に関係するため、勝手に変更しない。

## 9. 実装原則

### 9-1. 制約追加

制約追加時は以下を必ず確認する。

- ハード制約かソフト制約か
- 既存制約と論理衝突しないか
- 既存 objectiveWeights と互換があるか
- 大型連休など現実運用で infeasible になりすぎないか

### 9-2. 目的関数追加

- ペナルティ量と重み係数を混同しない
- 既存目的関数との優先順位関係を崩さない
- 「新しい名前の項目を追加」より「既存項目との互換維持」を優先する

### 9-3. 未記載仕様

- 未記載挙動を勝手に数式化しない
- TODO / 保留 / 確認事項として扱う

### 9-4. solver の責務

- solver 内に DB 事情や API 表示事情を持ち込まない
- optimizer は純粋な最適化ロジックを中心とする

## 10. テスト / 動作確認方針

制約変更時は、少なくとも以下を意識して確認する。

- 新制約が効いたケース
- 既存制約と衝突しないケース
- infeasible になるケース
- 月跨ぎ条件が効くケース
- 土日祝や大型連休での挙動
- target / score_balance への副作用

可能なら、小さい人工ケースで「この制約が効いた結果か」を説明できる状態にする。

## 11. backend / frontend との境界

- frontend は UUID ベース表示と入力整形を担当
- backend は API, schema, 永続化, 変換を担当
- optimizer は数理モデルを担当

他レイヤーで解決すべき問題を optimizer に押し込まない。

## 12. 実装時ルール

- 数理モデルは簡潔に保つ
- 既存制約を壊す大改修は避ける
- 名前互換を重視する
- 制約追加時は確認観点も併記する
- 質問や選択が必要な場合は、選択肢ごとの差分・影響範囲・推奨案を短く整理する
